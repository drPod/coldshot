## Usage philosophy

The user sends one cold email at a time. Speed doesn't matter.

## Sumble API — undocumented behaviors

These are not in the official docs and were discovered through testing.

**User-Agent blocking**: The API returns 429 for requests with the default `python-httpx` user agent. Set a custom User-Agent (we use `cold-sales/0.1.0`).

**Job level filter values**: The API uses `CXO` (not `C-Suite`), `VP`, `Director`, `Head`, `Manager`, `Senior`. The docs say "use values from the Sumble web app" but don't list them.

**Job function values are broad**: A CTO may be labeled `Executive`, not `Engineering`. Don't filter by job function when looking for technical leadership — filter by level only.

**Response field names differ from docs**: `FindPeopleResponse` uses `people_count` and `people_data_url`, not `total` and `source_data_url` as the docs suggest. Other response models (`FindJobsResponse`, etc.) use `total` and `source_data_url` as documented.

**Person page URL format**: The `url` field in API responses returns a tracking link (`https://sumble.com/l/person/...`). The actual person page is at `https://sumble.com/orgs/{org_slug}/people/{person_id}`.

**`technology_categories` values**: "Artificial Intelligence" is NOT a valid category for `organizations/find`. The docs don't list valid values. Use `technologies` with specific tech names instead (e.g., "LangChain", "OpenAI", "LLM").

**Email lookup is web-only**: There is no public API endpoint for email. The web UI uses `POST https://sumble.com/api/people/email` which requires a login JWT, an `X-User-Id` header, and an `X-Level` per-request signature generated by client-side JavaScript. This cannot be called programmatically from the API key.
